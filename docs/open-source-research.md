# Open-Source Video Project Research

Date: 2026-06-09

## Selection Principle

Maximize reuse of mature open-source projects, but keep video quality as a first-class constraint. The right shape is not a from-scratch editor. It is a thin orchestration layer that standardizes specs, profiles, review, render adapters, quality gates, and publishing handoff.

## Recommended Core

### 1. Remotion

Use Remotion as the default high-control rendering backend for componentized short videos.

Why it fits:

- Official docs position it as a way to make videos programmatically with React and create real MP4 videos.
- `@remotion/renderer` exposes `renderMedia()` for server-side/programmatic rendering.
- It supports browser preview, frame scrubbing, React composition, audio, and programmatic props.
- The existing `zhibo/videos/remotion-lab` already has an 84-composition reviewable element gallery, so the project has local Remotion investment.

Tradeoff:

- Remotion is excellent when the authoring system is React. It is heavier for agent-authored plain HTML and needs care around licensing for company-scale use.

Use it for:

- `zhibo` high-density AI workflow shorts.
- Motion graphics, code/browser frames, charts, overlays, mascot layers, platform-specific variants.
- Any video where reusable React components and timeline control matter.

Sources:

- https://www.remotion.dev/
- https://www.remotion.dev/docs/renderer/render-media
- https://www.remotion.dev/docs/captions

### 2. HyperFrames

Use HyperFrames as the default agent-authored HTML-to-video backend and as the strongest open-source reuse candidate for this repo.

Why it fits:

- HyperFrames is designed around plain HTML/CSS/JS compositions, deterministic rendering, headless Chrome, and FFmpeg.
- It is explicitly agent-friendly: agents can write HTML, preview it, lint it, and render MP4.
- It supports seekable web animations and a catalog-style component model.
- The local `zhibo/videos/hyperframes-engine` checkout already contains a full OSS engine with CLI, core, engine, producer, studio, player, shader transitions, and cloud packages.

Tradeoff:

- It is newer than Remotion. Keep Remotion as the stable path for production until HyperFrames output quality and operational reliability are verified on our real profiles.

Use it for:

- HTML slide/promo/story compositions generated by agents.
- Fast iteration where React scaffolding would slow creation.
- Deterministic CI rendering and frame hashing experiments.

Sources:

- https://hyperframes.video/
- https://hyperframes.video/docs
- https://hyperframes.heygen.com/packages/engine
- Local: `/Users/zhen.liu/projects/zhibo/videos/hyperframes-engine`

### 3. FFmpeg

Use FFmpeg as the shared media assembly and post-processing backend.

Why it fits:

- Existing `zhibo` already depends on FFmpeg for merging, subtitle burn-in, overlays, cover extraction, and anti-fingerprint variants.
- `novel-pipeline` already uses FFmpeg for Ken Burns still-frame clips and concat output.
- FFmpeg keeps the final encode path independent of a specific authoring engine.

Use it for:

- Concatenation and muxing.
- Audio normalization and ducking.
- Subtitle burn-in with libass where required.
- Platform variants and visual/audio fingerprint randomization.

### 4. Whisper / faster-whisper

Use faster-whisper style transcription for recording workflows and speech-to-caption alignment.

Why it fits:

- `zhibo` already has `videos/_assets/transcribe.py` using faster-whisper + VAD.
- Recording-mode subtitles are a real requirement, not a hypothetical feature.

Use it for:

- Recording-mode transcript generation.
- Alignment checks between narration and captions.
- Post-render QA: detect missing/late captions and obvious speech gaps.

### 5. OpenTimelineIO

Use OpenTimelineIO as an export target, not as the main renderer.

Why it fits:

- OTIO is an open timeline interchange format for editorial data.
- It gives an escape hatch to DaVinci Resolve / Premiere / other NLE workflows without turning our project into a GUI editor.

Use it for:

- Manual editor handoff.
- Debugging timeline structure.
- Future import/export compatibility.

Sources:

- https://otio-website.vercel.app/

## Projects To Avoid As Core

Browser GUI editors and cloud-template APIs are useful references but should not become the core:

- Creatomate: useful as fallback cloud renderer, but it is not open source and adds per-render dependency.
- CapCut/Submagic style APIs: useful for benchmarking captions and viral styling, but not a reusable OSS base.
- Full NLE clones: often optimize for interactive editing, while our need is automated batch production with review gates.

## Decision

Build `video-maker` as a portable spec and orchestration project:

1. Keep video definitions engine-neutral.
2. Implement `renderAdapter = remotion | hyperframes | ffmpeg`.
3. Keep `publisherAdapter = zhibo-sau | side-hustle-postiz | manual`.
4. Put quality gates in front of publish, independent of renderer.
5. Allow both existing projects to migrate gradually through file-based protocols.
